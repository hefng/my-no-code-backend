## 每个文件的主要修改内容

### VueProjectCodegenService.java（新增）

定义 Vue 工程化项目 AI 生成服务接口，使用 `TokenStream` 返回类型，通过 `onPartialThinking` / `onPartialResponse` 回调分别获取深度推理模型的思考过程和最终答案，避免依赖 `<think>` 标签字符串解析。

### VueProjectCodegenServiceFactory.java（新增）

每次请求新建服务实例（因为 `VueProjectFileSaveTool` 持有 `appId` 状态），注入 `reasoningStreamingChatModel` 和 `VueProjectFileSaveTool`，让 AI 可以直接调用工具写文件。

### VueProjectFileSaveTool.java（新增）

通过 LangChain4j `@Tool` 注解暴露给 AI 的文件写入工具。AI 在生成每个文件后主动调用此工具，彻底替代"后端解析代码块"方案。包含路径穿越安全校验，防止写入项目目录之外的文件。

### ReasoningStreamingChatModelConfig.java（新增）

手动注册名为 `reasoningStreamingChatModel` 的 Bean，通过 `application-local.yml` 中的 `ai.reasoning-model.*` 配置项注入推理模型参数（baseUrl、apiKey、modelName、returnThinking 等），与 langchain4j auto-config 创建的普通流式模型并存。

### SseEventTypeEnum.java（新增）

定义 SSE 事件类型：`thought`（思考过程）、`answer`（最终答案）、`done`（流结束信号），统一前后端事件命名约定。

### AiCodegenService.java（修改）

将 `generateHtmlStream` 和 `generateMultiFileCodeStream` 的返回类型从 `Flux<String>` 改为 `TokenStream`，以支持 `onPartialThinking` 回调获取深度推理模型的 `reasoning_content`。

### AiCodeGeneratorServiceFactory.java（修改）

将注入的 `streamingChatModel` 改为 `reasoningStreamingChatModel`，使 HTML / MULTI_FILE 生成也使用推理模型，统一享受深度思考能力。

### AiCodegenServiceFaced.java（重构）

核心重构：
- 新增 `generateCodeStream(TokenStream, ...)` 方法，通过 `Sinks.Many` 将 `TokenStream` 的 `onPartialThinking` / `onPartialResponse` 回调桥接为 `Flux<String>`，每个元素格式为 `{"event":"thought"|"answer","d":"内容片段"}`
- 新增 `generateVueProjectStream(...)` 方法，逻辑与上述相同，但使用 `VueProjectCodegenServiceFactory` 构建服务
- 新增 `buildEventJson(SseEventTypeEnum, String)` 工具方法，统一构建 SSE 事件 JSON
- 原 `processCodeStream(Flux<String>, ...)` 方法被删除，由上述新方法替代

### CodegenTypeEnum.java（修改）

新增 `VUE_PROJECT("vue_project", "Vue工程化项目生成")` 枚举值；同时修正 `MULTI_FILE` 的描述文字（原为"管理员用户"，属于复制粘贴错误）。

### AppController.java（修改）

- `chatToGenCode` 接口新增 `@RequestParam String codegenType` 参数
- 重构 SSE 事件路由：从 chunk 中解析 `event` 字段，Vue 工程化的 `thought`/`answer` 事件直接透传为 SSE event 名称；HTML/MULTI_FILE 的 chunk 降级为 `message` 事件，保持向后兼容
- 对话历史收集逻辑改为只提取 `d` 字段内容，过滤掉 `event` 元数据，避免将 JSON 结构存入历史记录

### AppServiceImpl.java（修改）

- 新增 `codegenType` 参数处理：若前端传入合法的 `codegenType`，校验后覆盖应用原有类型并持久化到数据库（仅在类型变化时执行 DB 写操作）

- 新增 `CodegenTypeEnum.getByType()` 调用，统一枚举解析

- 新增 `spring.mvc.async.request-timeout: -1`，解决 SSE 长连接被 Spring MVC 默认超时（30s）断开的问题

## 修改的原因和目的

### 核心目标：支持 Vue 工程化项目生成

原有系统只支持生成单文件 HTML 和多文件（HTML+CSS+JS）代码，无法生成完整的 Vue 工程化项目（含 `package.json`、`vite.config.js`、多个 `.vue` 组件等）。本次变动引入了第三种生成模式 `VUE_PROJECT`，通过以下两个关键设计实现：

1. **AI @Tool 直接写文件**：不再依赖后端解析 AI 输出的代码块，而是让 AI 在生成每个文件后主动调用 `VueProjectFileSaveTool.saveFile()`，文件保存的时机和内容完全由 AI 决策，更灵活可靠。

2. **深度推理模型 + thought/answer 事件拆分**：引入独立的 `reasoningStreamingChatModel` Bean，开启 `returnThinking`，通过 LangChain4j `TokenStream` 的 `onPartialThinking` 回调获取思考过程，前端可实时展示"AI 正在思考..."，提升用户体验。

### 修复 SSE 长连接超时问题

Spring MVC 默认异步请求超时为 30 秒，导致复杂代码生成任务（尤其是 Vue 工程化项目）在生成过程中被强制断开。通过设置 `request-timeout: -1` 彻底解决。

### 统一 SSE 事件格式

原有 HTML 生成的 SSE 数据格式为 `{"d":"代码片段"}`，本次统一为 `{"event":"thought"|"answer","d":"内容片段"}`，前端通过 `event` 字段区分思考流和答案流，同时保持向后兼容（无 `event` 字段时降级为 `message` 事件）。

### codegenType 动态切换与持久化

前端在发起生成请求时可以传入 `codegenType` 参数，后端校验后覆盖应用原有类型并持久化，确保下次进入应用时预览 URL 和生成逻辑保持一致，避免类型漂移。

## 可能的影响范围

- **前端 SSE 消费逻辑**：需要适配新的事件格式，增加对 `thought` 和 `answer` 事件的处理；原有 `message` 事件仍然兼容，HTML 生成不受影响。
- **`chatToGenCode` 接口**：新增必填参数 `codegenType`，前端调用时需要传入（可传 `html` 保持原有行为）。
- **Redis 数据库**：`application.yml` 中 Redis database 切换为 db1，部署时需确认 Redis 实例 db1 可用，已有 db0 数据不受影响。
- **推理模型配置**：`application-local.yml` 需新增 `ai.reasoning-model.*` 配置项，否则 `ReasoningStreamingChatModelConfig` Bean 初始化会失败。
- **Vue 工程化文件目录**：生成的文件保存在 `{CODEGEN_DIR}/vue_project_{appId}/` 目录下，部署环境需确保该目录有写权限。
